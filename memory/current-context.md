# Current Context

- Warehouse designer + Magazyn: page-level vertical scroll is reduced via a complete flex `min-h-0` chain (`MainLayout` → `PageLayout` `fillHeight` → content) and sidebar refactors (`TopProductsSidebar`, `ProductLocatorSidebar`); map scroll stays on the map column.
- Magazyn + layout designer horizontal layout: left rail `260px`, right panels `320px`, center `flex-1 min-w-0 min-h-0 overflow-auto` so the canvas shrinks instead of pushing sidebars off-screen.
- Designer vertical fill: main app column is `overflow-hidden` on designer routes; `PageLayout` fillHeight + `h-full`/`max-h-full`/`flex-1`/`min-h-0` chain; scroll only in map/canvas `h-full w-full overflow-auto` wrappers.
- Magazyn: sidebar product click toggles `selectedProductId` and bin highlights on the map (UUID overlays in `RackLayer`); Ctrl+click still opens product.
- Destructive confirmations: clear-rack modal shows rack label + assignment count; single-location remove modal shows location + optional product name (copy only).
- Recent focus: route visualization was refactored toward a stop-first model.
- Route path is now high-clarity: strong solid blue stroke (opaque), repeated direction arrows along the path, and a subtle animated flow overlay.
- Numbered step badges render at **aisle pick points** (`getRackPickPointCell` + `getRackRouteWaypoint`, offset from user aisle width) with a small nudge into the aisle; `routeStops` positions feed path + badges.
- Path now also renders `1..N` markers projected directly on the route polyline, and `RouteStopLayer` draws short connector lines from each stop to the corresponding rack edge.
- Sidebar stop selection highlights the corresponding rack and marker.
- Current route visualization work is frontend-only; backend route logic remains unchanged.
- Route markers were strengthened for human picking: solid blue stop badges with white numbers, a distinct green START marker, and stronger current-stop emphasis.
- Path styling was further demoted to light, thin dashed guidance so users can follow stop order without relying on the path.
- In the Warehouse Layout Designer, route visualization should show the full route for planning and analysis: all stops remain visible, markers use a uniform style, and step-by-step picker states should not be used.
- The Warehouse Layout Designer now has a `showRoute` visibility toggle that hides or shows `PathLayer` and `RouteStopLayer` without changing route data or calculations.
- In planning view, each stop marker should communicate both order and rack identity directly on the map; the route path should only provide subtle directional guidance.
- For route preview readability, the path should feel continuous into the stop markers: use stronger solid path styling with direction cues, while keeping all stops visible and unchanged structurally.
- Path rendering now includes a frontend-only simplification pass: snap tiny deviations back to aisle-like horizontal/vertical movement, remove redundant collinear points, and collapse small jogs before drawing.
- Route preview now also supports an explicit high-visibility overlay mode in `PathLayer`: solid blue route line, white halo underlay, repeated arrow glyphs, and console logging for `pathPoints`, `showRoute`, and render state debugging.
- In `WarehouseCanvas`, route layers should render in this order for visibility: `RackLayer`, then `PathLayer`, then `RouteStopLayer`.
- In route preview mode, `PathLayer` now visually constructs the main route from `routeStart` plus `routeStops.position` using simple orthogonal one-turn segments instead of rendering the raw backend node chain.
- `buildVisualRoutePath` must always emit explicit visible geometry per leg in route mode: `start -> turn/midpoint -> end`, with runtime logging for generated point count and first-segment points.
- Magazyn/read view reuses the exact same shared `PathLayer` logic as layout; only `RouteStopLayer` marker placement changes there so stop numbers sit near the path instead of on rack centers.
- In Magazyn product-highlighting mode, `RackLayer` now dims non-matching racks while keeping selected racks highest priority and product-matched racks visually prominent with stronger outline/fill.
- In Magazyn product-locator mode, the active product now also produces a frontend-only `rackQuantities` map so highlighted racks can show per-rack stock quantity badges without changing existing highlight logic.
- Layout identity refactor step 1 is now in place: backend racks expose a persisted `uuid`, bins expose a persisted `location_uuid`, older rows are backfilled on layout load, and the frontend stores/preserves those identity fields without introducing diff-based save behavior yet.
- Layout refactor step 2A is now partially implemented in backend save flow: racks are matched by `uuid` and updated in place instead of always being recreated, new racks are inserted with a generated `uuid`, removed racks are deleted by rack row only, and the old destructive rack path remains available behind `RACK_IDENTITY_SAVE_ENABLED` as a fallback. Existing bins are not identity-migrated yet.
- Layout refactor step 2B is now implemented in backend save flow: bins are matched by `location_uuid`, updated in place without changing that UUID, new bins get a generated UUID only when needed, removed bins are marked inactive instead of deleted, `Location` rows now have their own `location_uuid` and are synced by UUID rather than label, and removed racks are also hidden via `is_active` so their bins and linked locations remain preserved.
- Soft-delete hardening is now in place: `Rack`, `Bin`, and `Location` all default to active-only reads through a global SQLAlchemy session filter, schema upgrades add `is_active` plus activity indexes, and only the layout save/sync path opts into `include_inactive=True` so inactive rows no longer affect UI, routing, graph assignment, or product-location queries.
- Layout refactor step 2C part 1 is now implemented for aisles: aisle persistence uses stable DB `id` as identity, frontend already round-trips that `id`, backend save now upserts existing aisles by `id`, creates new aisles when no `id` is present, and safely deletes only aisles missing from the payload instead of recreating the whole aisle set on every save.
- Layout refactor step 2C part 2 is now implemented for `visual_elements`: backend layout payload accepts the frontend visual element shape, `warehouse_layouts` stores them as JSON in `visual_elements_json`, `save_layout` overwrites that JSON safely, and `get_layout` returns `visual_elements` so canvas-only elements persist across reloads.
- Layout API step 3 is now implemented: `GET /warehouse/layout` returns a unified read model with `{ layout, special_locations }`, the old standalone special-location endpoints still exist, `WarehouseDesigner.loadLayout()` now hydrates both layout and special locations from one request, and other frontend `/warehouse/layout` readers accept either the new wrapped response or the old flat shape during transition.
- Rack persistence has been hardened against invalid payloads: save now validates rack footprint fields before insert, skips new racks with missing `x`/`y`/`width`/`height` instead of crashing, preserves existing DB values when partial updates contain `None`, and emits warning logs for skipped racks and partial rack updates.
- Storage location consistency is now being synchronized from bins: `StorageLocation` has `location_id` and `is_active`, `save_layout()` calls a new `_sync_storage_locations_from_bins()` pass after rack/bin/location processing, active bins are reconciled to exactly one active storage-location record via `location_uuid -> Location`, and inactive bins now deactivate related `StorageLocation` rows instead of deleting them.
- Rack type support in frontend is now fully wired for creation/save/reload: racks always carry `rack_type` (`warehouse`/`store`), template-based placement resolves `template.rack_type ?? "warehouse"`, manual placement has a WH/STORE selector, and rack tiles display a small WH/STORE badge.
- Rack side-view UX now includes storage-type visualization: each slot shows a top-right storage-type icon, clicking the icon toggles same-type highlighting, and `WarehouseLegend` rack-detail mode shows a new "Typy lokalizacji" section with only types used in the currently displayed rack.
- Rack type mode is now a real working filter in the designer: sidebar templates and "used templates" are filtered by active `manualRackType`, magazyn dashboard template usage is filtered by that same type, and layout generation now filters custom templates and applies the active rack type for preset-generated racks.
- Switching active rack type now resets dependent template state (`rowToolTemplate` and magazyn `selectedTemplateId`) to avoid cross-type stale selections.
- Warehouse designer top action bar now contains the primary `Zapisz układ` action (layout mode only), and the duplicate bottom save button was removed from the sidebar list panel.
- `TemplateCreator` rack preview now uses a single physical-proportion mode (no structure/names tab): levels are height-proportional, slot widths are rendered from per-level width share, orange low-opacity travers lines separate levels, and compact cells hide detail lines when too narrow.
- Internal rack-structure edits now support template variants in `WarehouseDesigner`: when a rack linked to a template gets a different structure (levels/locations-per-level), save flow prompts to create a new variant and relinks only that rack to the new variant (`variant.templateId = baseTemplate.id`) while keeping base dimensions.
- Picking-route UX refactor is now mode-less in `WarehouseDesigner`: route selection state is `routeRackIds` (ordered) and rack clicks directly toggle inclusion without any `routeMode` flag.
- Floating bottom-right route card was removed; route planning UI now lives in the existing right `RackPropertiesSidebar` under a new "Trasa kompletacji" section with ordered racks, distance, clear, and optimize actions.
- Route planning affordances: `routeMode` in mouse handlers matches `isRouteActive` (empty route still accepts clicks); rack `onRackClickPassthrough` is disabled in route mode to prevent click-after-mousedown double toggle; toolbar and sidebar copy reflect active planning state; `isRoutePlanningMode` drives rack hover + pointer on the layout canvas.
- Route visualization: stop positions use aisle access points (`getRackAccessPointCell`); path builds from START through those points to nearest packing station (`collectPackingCentersCells` + `pickNearestPackingCell`); `RouteStopLayer` shows only numbered stops + PACK + START; `PathLayer` omits duplicate path markers in route mode.
- Canvas route visibility is now automatic: route path renders whenever at least 2 racks are selected (`showRoute` derived from `routeStops.length >= 2`), and route order badges are drawn directly on racks via `RackLayer`.
